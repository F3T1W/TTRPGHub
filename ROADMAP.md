## Северная звезда

Сделать так, чтобы русскоязычному игроку/мастеру НРИ не нужно было уходить на сторонние
сайты ни для чего: вкат в хобби, изучение правил, билдостроение, поиск пати, ведение игр.
Один сайт закрывает весь цикл от «первый раз слышу про D&D» до «веду кампанию третий год».

**Смена приоритета (актуально с этого момента):** основная система — не D&D5e, а **Pathfinder 2e**.
Конкретная цель — заменить Foundry VTT для онлайн-групп, которые сейчас гоняют сессии через
удалённый рабочий стол (Radmine и т.п.), поднимая Foundry на чужой машине. Это высокая планка:
Foundry даёт автоматизацию правил (степень успеха, MAP), карты с сеткой/туманом войны/шаблонами
зоны поражения, и огромный компендиум контента. Разрыв и план закрытия — см. «Фаза H» ниже.

## Архитектурный принцип: системная независимость

Это ключевое решение, определяющее всю дальнейшую работу с правилами и билдером.

**Сейчас:** `Dnd5eSpell`, `Dnd5eMonster`, `Pf2eSpell`, `Pf2eMonster` — отдельные сущности
с дублирующейся структурой на каждую систему. Добавление новой официальной системы или
кастомной самописной требует копипасты целого вертикального среза (Domain → Persistence → API → Web).

**Куда идём:** единая системно-независимая модель справочника:

```
GameSystem (справочник систем: "dnd5e", "pf2e", "custom:<slug>")
├── Id, Slug, Name, IsOfficial (bool), CreatedByUserId (null для официальных)

RuleEntry (заменяет Dnd5eSpell/Monster и т.д. — одна сущность на любой тип контента)
├── Id, SystemId (→ GameSystem)
├── Category: enum { Spell, Monster, Class, Race, Feat, Condition, Equipment, Background, Rule }
├── Slug, Title, Summary, ContentMarkdown
├── Stats: JSONB (гибкая схема под категорию — для Spell одни поля, для Monster другие)
├── Tags: string[]
└── IsHomebrew: bool (отличает официальный контент от пользовательского)
```

Почему JSONB для `Stats`, а не строгая EF-схема под каждую категорию: количество комбинаций
(категория × система) растёт неограниченно при добавлении кастомных систем, а строгая
реляционная схема под каждую — это та же проблема дублирования, только глубже. Гибкие поля
(`Stats` как JSONB + типизированный C#-DTO поверх для известных официальных категорий)
дают системную независимость без потери строгой типизации там, где она нужна (D&D5e/PF2e).

**Переходный путь (не революция, а эволюция):**
1. Новый `RuleEntry` + `GameSystem` живут рядом со старыми `Dnd5eSpell`/`Pf2eSpell` — ничего не ломаем
2. Пишем миграционный скрипт, переносящий существующие D&D5e/PF2e данные в `RuleEntry`
3. Когда фронтенд и API переключены на `RuleEntry`, старые таблицы выпиливаем
4. Homebrew-хаб (он уже умеет произвольный контент) становится фронтендом для создания
   кастомных `GameSystem` + `RuleEntry` — переиспользуем существующую инфраструктуру вместо
   написания новой с нуля

      `RuleEntry` содержат уже переведённые данные — перевод происходит один раз при импорте,
      не на горячем пути запросов. Slug остаётся на латинице от оригинального английского имени
      (стабильные URL), отображаемые названия — на русском
- [x] Кастомные системы: `POST /api/v1/rules/systems` создаёт `GameSystem.CreateCustom` (слаг —
      транслитерация названия + де-дубликация суффиксом `-2`, `-3`...), `POST/PUT/DELETE
      /api/v1/rules/{systemSlug}/{category}` управляют `RuleEntry` внутри своей системы.
      Построено параллельно старому Homebrew-хабу (форумный "один блоб текста"), не поверх
      него — Homebrew не имеет структурированной `Category`/`StatsJson`/Markdig-рендера и не
      может отдавать данные в унифицированный `/reference/{system}/rules/{category}` без
      переписывания. Авторизация: владеет записями тот, кто `GameSystem.CreatedByUserId`
      (per-entry владения нет, `RuleEntry` не хранит своего автора — осознанное упрощение);
      `Error.Forbidden()` → HTTP 403 (заодно исправлена недостающая мапа в `ToProblem()` —
      раньше 403 не существовал, всё falls back на 400). Официальные системы (`IsOfficial=true`)
      неприкосновенны — попытка добавить/изменить/удалить запись в dnd5e/pf2e даёт 403.
      UI: `/reference/create-system` (форма создания), кнопка «Своя система» на `/reference`,
      кастомные системы рядом с официальными карточками с бейджем «Homebrew · моя»/«Homebrew»;
      `/reference/{system}/rules/{category}/create` и `.../{slug}/edit` — общая форма (категория,
      название, краткое и полное описание в Markdown, `StatsJson` как сырой JSON-textarea,
      теги через запятую); кнопки редактировать/удалить на `Detail.razor` видны только владельцу
      (`RuleEntryDetailDto.CanEdit`). Для кастомных систем список категорий на `List.razor`
      расширен до всех 9 (включая Spell/Monster/Equipment/Background) — у официальных систем
      навигация не меняется (там Spell/Monster живут на отдельных legacy-страницах `/spells`,
      `/monsters`). Проверено end-to-end curl + браузером: создание системы → транслитерация
      слага → добавление монстра → просмотр → 403 на чужую/официальную систему → правка
      владельцем → удаление владельцем → 404 после удаления

      и best-effort достаёт «что нового» из markdown-таблицы прогрессии класса
      (`CharacterAutomationCalculator.FindLevelFeatures`). UI: кнопка «Уровень» на странице
      персонажа. Полный ручной разбор фитов/слотов заклинаний за уровень — не сделан,
      это следующий шаг данной фичи, если понадобится глубже
- [x] Мультикласс-калькулятор — `/tools/multiclass`, standalone (не трогает `Character`,
      чтобы не рисковать доменом): считает суммарный уровень, бонус мастерства и пул костей
      хитов по правилам D&D5e мультикласса (первый уровень первого класса — максимум кости,
      остальные — среднее)
- [x] Мастер для PF2e — `CreateCharacterFromRulesCommandHandler` теперь ветвится по `SystemSlug`
      (`"pf2e"` → `Pf2eCharacterAutomationCalculator`, иначе прежний D&D5e-путь). Новый калькулятор
      **сознательно упрощён**, не претендует на полную точность PF2e-правил — задокументировано
      прямо в коде и в подсказке мастера персонажа:
      - Буст характеристик: применяются только явные пары предок/класс (`boost_codes`/`flaw_code`
        предка + `key_ability_codes` класса, если код один) по стандартной формуле PF2e
        (+2 если текущее значение <18, иначе +1; провал предка −2). Свободный выбор буста
        (`"ANY"` у Человека/Полуорка, второй ключевой атрибут у Бойца/Следопыта) **не
        применяется автоматически** — игрок поднимает нужную характеристику вручную до отправки формы
      - HP = HP предка + уровень × (HP класса за уровень + мод. Телосложения) — верная формула PF2e
      - AC = 10 + мод. Ловкости + (уровень + 2) — предполагает "тренированное" владение обороной
        без доспеха/щита; ranks владения (untrained/trained/expert/master/legendary) по отдельным
        показателям не моделируются, потому что `Character`-сущность хранит один плоский
        `ProficiencyBonus` в D&D5e-стиле (уровень→бонус по таблице 5e) — полная переработка модели
        персонажа под PF2e ranks не входила в скоуп этой итерации
      - `Character.SavingThrowProficiencies` остаётся пустым списком для PF2e-персонажей
        (вместо того чтобы подделывать несуществующее бинарное владение спасброском)
      - `Pf2eRulesSeeder` дополнен структурированными кодами (`key_ability_codes`,
        `boost_codes`, `flaw_code` — рядом с прежним читаемым текстом `key_ability`/`boosts`/`flaw`
        для отображения в справочнике)
      - UI: `/characters/create-wizard` — добавлен селектор системы (D&D 5e / Pathfinder 2e),
        переключение перезагружает списки рас/классов из справочника; при выборе PF2e показывается
        предупреждение об упрощениях
      - Проверено end-to-end curl (Дварф-Варвар 1 уровня: буст CON+2/WIS+2, провал CHA−2, буст
        класса STR+1 — все числа сошлись; Человек-Волшебник 5 уровня: свободный буст предка не
        применён, буст класса INT+1 применён, HP=43 и AC=19 математически верны) + браузером
        (переключение системы подгружает списки предков/классов PF2e)
- [x] **Проблема качества данных с терминологией — исправлено**: причина была в
      `JsonTranslationHelper`, который переводил вообще все строковые поля, включая игровую
      нотацию костей (`hit_dice`/`damage_dice`, например "1d12") — Google Translate иногда
      портил латинскую `d` на кириллическую `д`, из-за чего `HitDieRegex` не матчился и HP
      автоматики тихо считался по дефолтной d8. Добавлен параметр `excludeKeys` — эти поля
      больше не переводятся при импорте. Плюс одноразовый SQL-проход нормализовал уже
      занесённые данные: капитализация первой буквы у переведённых названий и унификация
      "Конституция"/"Телосложение" → "Телосложение" везде. Полный переимпорт D&D5e данных
      прогнан и проверен (класс Варвар: `hit_dice` = "1d12", левел-ап 4→8 даёт корректные 85 HP)

### Осознанно не в списке: макросы/скрипты
Foundry позволяет GM писать произвольный JS-код, исполняемый в контексте сессии. В вебе с общим
клиентским рантаймом на всех участников это прямой путь к XSS/RCE — один злонамеренный или просто
кривой макрос ГМ может выполниться в браузере игрока через SignalR-синхронизацию состояния.
Правильная замена — не "макросы" 1:1, а набор безопасных пресетов (как уже сделано для
проверок/атак из PF2e-листа в H.7/H.8), расширяемый по мере необходимости, а не открытая
песочница выполнения кода.

## Фаза J — От «карта + справочник» к полноценной замене Foundry

Гэп-анализ после завершения Фазы I и реального импорта контента (спеллы/фиты/снаряжение/бестиарий).
Справочник и стол готовы, но правила не исполняются автоматически, а бой ведётся вручную. Пункты
упорядочены по влиянию на ощущение «полноценной замены» в реальной игре.

### J.1 — Автоматизация правил (rule elements)
- [x] **Первый срез сделан и проверен end-to-end**: безусловные (без `predicate`) числовые
      `FlatModifier`-правила из данных фитов (`land-speed` и селекторы навыков/AC/perception)
      извлечены в `pf2e-feats.json` (поле `modifiers`) и хранятся в `StatsJson` каждой записи
      справочника (`Pf2eContentSeeder.BuildFeats`)
- [x] Добавлен пакетный эндпоинт `POST /api/v1/rules/{system}/{category}/batch` +
      `IRuleEntryRepository.GetBySlugsAsync` — лист персонажа одним запросом подтягивает
      структурированные данные всех своих фитов вместо N отдельных обращений
- [x] `Pf2eFeat` (Character.Pf2eStatsJson) получил необязательное поле `Slug`: при вводе
      названия фита на листе персонажа (`Detail.razor.cs.UpdateFeat`) идёт точный поиск
      (без учёта регистра) по каталогу — при совпадении фит связывается со справочником
      (иконка-цепочка в режиме просмотра), при отсутствии совпадения остаётся обычным текстом
      без автоматизации — деградация, а не ошибка
- [x] Связанные фиты автоматически суммируют `land-speed`-модификаторы в эффективную скорость
      персонажа, отображаемую как `35 фт (30+5)` — проверено вживую (фит Fleet, +5 футов)
      через реальный аккаунт/персонажа в браузере, не только через API
- [x] **Доведено до практического предела на статичном листе персонажа** (без combat tracker):
  - **Движок предикатов** (`Pf2eLookups.PredicateEvaluator`) вычисляет условия PF2e (`predicate`
    из данных Foundry) против статичных фактов персонажа: строки-термы (AND по массиву),
    `{or/and/not/nor/nand: [...]}`, числовые сравнения `{gte/lte/gt/lt: [терм, число]}`.
    Известные факты: `feat:<slug>` для каждого связанного фита, `self:level`, `hp-percent`,
    `skill:<навык>:rank` (ранг владения 0–4 из `SkillRanks`). Нераспознанный терм (`action:*`,
    `target:*`, произвольные формулы) безопасно даёт `false` — модификатор не применяется вместо
    применения неверно
  - Извлечение фитов расширено с безусловных до **всех** `FlatModifier`-правил с числовым
    значением на известных селекторах (скорость/навыки/AC/perception) — `predicate` каждого
    правила переносится как есть и вычисляется на клиенте. Модификаторов с известными
    селекторами стало **172** (было 11), затрагивают **161** фит (было 11)
  - **GrantItem** реализован как отображение, не автодобавление: `grants` (имена
    предоставляемых айтемов, распарсенные из `uuid`) показываются под связанным фитом текстом
    "Даёт: …" — **469 фитов** несут такие данные. Проверено вживую: Duelist Dedication → "Даёт:
    Quick Draw"
  - **Состояния (Conditions) переимпортированы из реальных данных Foundry** (`pf2e-conditions.json`,
    43 состояния из `packs/pf2e/conditions`, ORC/OGL) вместо 27 вручную написанных — включают
    структурированные формулы (`{kind: "per_badge", multiplier: -1}` для "штраф = значение
    состояния", как у Frightened/Sickened/Clumsy/Enfeebled/Stupefied) вместо только текста.
    Данные готовы к использованию будущим combat tracker (J.2) — само применение состояния к
    броску/AC в бою не реализовано, т.к. пока нет боевого контекста, куда его применять
  - Skill/AC/perception/speed на листе персонажа показывают итог с разбивкой
    (`35 фт (30+5)`), пересчитывается при любом изменении списка фитов
  - Пакетная выборка расширена (батч-эндпоинт из первого среза без изменений)
  - **Честный итог проверки данных**: из 172 извлечённых модификаторов только **11** (те же,
    что и в первом срезе) реально вычисляются в `true` на сегодняшнем наборе фактов — почти все
    предикаты в реальных данных PF2e завязаны на `action:*`/`target:*` (конкретное действие,
    конкретная цель броска), которых на статичном листе персонажа нет и не может быть без
    системы бросков/боя. Движок предикатов корректно построен и протестирован (or/not/gte и
    т.д.), но раскроет весь потенциал только после появления боевого контекста (J.2) — тогда
    добавление фактов `action:<slug>`/`target:trait:<x>` в момент броска даст немедленный прирост
    без изменения самого движка
- [ ] Аналогичного разбора модификаторов/предикатов для снаряжения (руны, доспехи) и монстров —
      структурированного автоматического парсера по-прежнему нет. Частичная замена появилась в
      Фазе P: `Pf2eStatsModel.CustomRollOptions` — эскейп-хэтч, где GM/игрок вручную вписывает
      нужный roll option строкой, покрывает практический случай без структурного парсинга

### J.3 — Освещение сцены (lighting), отдельно от стен — сделано и проверено вживую
- [x] **Источники света**: `GameSession.LightsJson` (JSON-массив `{x,y,brightRadiusFeet,

### M.1 — Импорт своего PDF-приключения в приватный контент сессии ✅
- [x] **Разбор лицензионного вопроса** (K.6 упирался в лицензии Paizo на готовые компендиумы):
      ГМ, показывающий партии купленный контент за своей игрой — это нормальное использование
      покупки (как чтение вслух из бумажной книги), а не распространение. Риск — если бы
      импортированный контент попадал в общий межпользовательский слой (как `RuleEntry`),
      где User B мог бы увидеть/скопировать то, что купил и импортировал User A: это уже было
      бы распространением чужого копирайта платформой. Поэтому контент строго привязан к
      `GameSession` конкретного ГМ — видим только ему и участникам именно этой сессии, не
      попадает в общий справочник. Foundry и другие VTT решают тот же вопрос тем же способом
      (мир/сцена ГМа, не общая библиотека).
- [x] **Парсинг PDF на сервере** — новый интерфейс `IAdventurePdfParser` (Application) +
      `PdfPigAdventureParser` (Infrastructure, пакеты `UglyToad.PdfPig` +
      `UglyToad.PdfPig.DocumentLayoutAnalysis`). Два нетривиальных момента, найденных при
      тестировании на реальном PDF (Paizo Free RPG Day one-shot, куплен пользователем):
      1. Обычный `Page.Text` перемешивает слова из соседних колонок двухколоночной вёрстки
         Paizo в нечитаемую кашу — нужен именно `ContentOrderTextExtractor` из пакета
         DocumentLayoutAnalysis, который восстанавливает порядок чтения геометрическим
         анализом колонок, а не наивным сравнением X-координат (наивная эвристика "midpoint
         страницы" пробовалась первой и давала заметно шумный текст — см. историю сессии).
      2. Картинки этого PDF используют цепочку фильтров `[/FlateDecode, /DCTDecode]` —
         `PdfPig.TryGetPng` не умеет их декодировать. `RawBytes` после `FlateDecode` — валидный
         JPEG-поток (DCTDecode = JPEG), просто PdfPig отдаёт его до инфлейта; вручную
         распаковываем zlib (`ZLibStream`) и сохраняем как обычный `.jpg`, декодировать сам
         JPEG не нужно.
- [x] **Текст → Journal, карты → превью для точечного создания сцен, не автоматически.**
      `ImportAdventurePdfCommand`: текст постранично уходит в `JournalEntry` (папка = имя
      файла + дочерние записи "Стр. N", черновики — ГМ решает, что публиковать), картинки
      выше порога размера (300×250) и без байтовых дублей (декоративные переиспользуемые
      иллюстрации — дедуп по SHA-256) загружаются в MinIO и возвращаются GM списком превью;

  - `RecomputeAuraEffectsAsync` (Table.razor.cs) — пересчёт с нуля при каждом движении токена
      (та же геометрия/математика, что у AOE-шаблонов J.5 и `Pf2eLookups.TokenDistanceFeet`, N.5):
      для каждого токена-монстра с аурой и каждого другого токена на сцене — входит ли цель в
      радиус. Триггеры пересчёта: SignalR `TokenMoved`/`TokenAdded`/`TokenRemoved`, локальный
      `OnTokenDragEnd`, и один раз при полной загрузке стола (`LoadTableStateAsync`) — не только
      при касте/входе в бой, а действительно в реальном времени при любом движении, как в Foundry
  - `_auraAppliedSlugsByToken` — отслеживает, какие слаги состояний на каком токене-цели именно
      аура наложила (не просто "состояние есть") — при выходе из радиуса снимается только то, что
      сама аура добавила, вручную наложенные ГМом состояния с тем же слагом не трогаются
  - Кеш аур по monsterId (`_monsterAuraCache`) — в отличие от `_selectedMonsterImmunities`
      (грузится только для выбранного/наведённого токена), тут нужны ауры **всех** токенов-монстров
      на сцене одновременно, кеш заполняется лениво при первом пересчёте после появления токена
  - Панель ГМа на выбранном токене монстра показывает "Ауры: ..." рядом с сопротивлениями/
      уязвимостями/иммунитетами
  - Проверено вживую: монстру выставлена аура "Испуган, 1, радиус 10 фт" → токен-цель на 50 фт
      (вне радиуса) — состояний нет → цель придвинута на 5 фт (в радиус) — `frightened` появился
      на токене автоматически через SignalR без ручного действия ГМа → цель отодвинута обратно на
      50 фт — состояние снято автоматически → консоль браузера чистая на всех этапах
  - Осознанно не в этом пункте: **массовое заполнение `AurasJson` для существующих монстров не
      сделано** — то же ограничение источника данных, что у `ImmunitiesJson` в N.4 (ауры в PF2e
      относительно редки у монстров, отдельного скрипта сопоставления с исходными данными Foundry
      нет); вся механика готова и заработает сразу при заполнении поля. Пересчёт "на старте хода"
      отдельно от пересчёта "при входе" не различается — аура пересчитывается при каждом движении,
      что фактически покрывает оба случая Foundry (вход в радиус ловится сразу же при движении)

### N.10 — Полноценный Rule Elements engine ✅

## Зависимости между фазами

```
Фаза A (Справочник 2.0) ──→ Фаза B (Билдостроение)
Фаза C (Онбординг) ──→ переиспользует RuleEntry из фазы A, но может стартовать параллельно
Фаза D (Тикеты) ──→ независима, требует только добавления ролей пользователей
Фаза E ──→ после A–D, не блокирует и не блокируется
Фаза L ──→ после H–K; L.1–L.2 закрывают критичный гэп автоматизации боя PF2e
```

Рекомендуемый порядок старта: **A → D (можно параллельно, т.к. независима) → B → C → E**.
